---

很多自学者在准备技术面试时，把所有精力放在刷题和背八股文上。

但这解释不了一个现象：为什么有时候技术问题全答上来了，还是挂了？

答案可能是——**技术面不只是考技术。** 面试官在技术面中同时考察的还有：你遇到不会的问题怎么反应、你怎么沟通你的思路、你面对压力时的状态。

对于自学者，技术深度不够扎实是事实，但软技能的差距是可以通过意识和方法来弥补的。

---

## 一、技术面面试官的隐性考察清单

面试官不会明说他在看什么，但他心里有一张清单：

### ✅ 显性考察（你准备了的）
- 这个技术点答对没有
- 代码能否正常运行
- 算法复杂度是否正确

### 🔲 隐性考察（容易被忽略的）
- **沟通效率**：他能不能在 30 秒内理解你说的方案？还是需要反复追问才能弄清楚你的意思？
- **问题拆解能力**：你遇到不会的问题时直接放弃，还是能拆解出自己能回答的部分？
- **面对反馈的反应**：他给提示后，你是抗拒防御，还是快速吸收并应用？
- **知识迁移能力**：一个没遇到过的问题，能不能用已有的知识推断出合理的答案？
- **信号诚实度**：你不会的东西是直接说"不知道"，还是试图含糊其辞？

很多自学转行的候选人，隐性考察的得分是负数。不是因为能力不够，是因为紧张、不自信、或者太想表现自己。

## 二、遇到不会的问题怎么办

这是技术面中最关键的时刻。你几乎一定会遇到几个不会的问题——面试官就是想看你怎么应对。

### ❌ 错误示范

**A 类型：沉默**
面试官问了一个问题，你沉默 30 秒，然后说"我不确定"。

→ 面试官看到的是：思考过程不透明，无法判断你的思维方式。

**B 类型：瞎编**
你不知道这个问题，但为了不丢面子，强行编了一个答案。

→ 面试官看到的是：不诚实，不可信。这是最致命的问题。

**C 类型：防御**
"这个问题我在实际中一般不会这么用"或者"这个 API 我不常用。"

→ 面试官看到的是：不愿意直面自己的知识盲区。

### ✅ 正确示范的三步法

**Step 1：确认问题**
"我先确认一下我的理解：你问的是不是 X 方向上的 Y 问题？"

这给了你 5 秒钟整理思路，同时展示你在做任何事之前先确认需求。

**Step 2：拆解已知**
"关于这方面，我了解的是 A 和 B。但你说的 C 我确实没有深入接触过。"

诚实承认不知道，但先展示你知道的上下文。这让面试官知道你不是完全空白。

**Step 3：推断或学习路径**
"不过根据我对 A 的理解，我推断 C 可能应该是这样的……我会从 D 这个方向去查资料来解决。"

这一步展示了你最核心的能力——**面对未知，你有推理和学习的方法论**。

## 三、沟通的艺术：思考过程&思路表达

### 边做边说的能力

写代码时、推导问题时保持"有声思维"——把你正在想的说出来。

不是自言自语，而是像一个"远程 pair programming"一样跟面试官沟通。

```markdown
"我先确认一下输入输出…… 这里输入是一个数组，期望输出是一个排序后的数组。
我想到的第一个方案是快速排序，平均时间复杂度 O(n log n)。
我需要一个 pivot 来分区…… 这里选中间值作为 pivot。
现在分区完了，左边都比 pivot 小，右边都比 pivot 大。
然后递归处理左右两边…… 好，代码写完了。我跑一下测试用例验证……"
```

### 用"概念"而不是"细节"先建立框架

回答复杂问题时，先用一句话说出整体思路，再展开细节。

❌ "我们用的是 PostgreSQL，因为……（开始讲数据库细节）"
✅ "这个问题我从三个方面来回答：数据存储、接口设计、缓存策略。首先数据存储上我选 PostgreSQL，理由是……"

### 追问和澄清是信号，不是弱点

很多自学者不好意思追问问题，担心显得自己"不懂"。

但事实上，在面试中追问澄清问题——尤其是需求不清晰的技术问题——是**成熟工程师的表现**。

"这个场景的并发量大概在什么级别？"
"这个接口的服务端是用什么技术栈实现的？"
"这个需求是倾向于优先性能还是优先可维护性？"

每一个追问都在传递一个信号：**我不只是会写代码，我会思考上下文。**

## 四、Pairing 和 Live Coding 中的表现

越来越多的技术面试会增加 Pair Programming 或者 Live Coding 环节。这一刻你面对的不是一个"考官"，而是一个"同事"。

### 几个关键原则

**别抢键盘。** 即使你知道答案，先跟面试官沟通你的思路再写。Pairing 中"先把想法说清楚再动手"是专业的表现。

**接受提示。** 如果面试官给了提示，说一句"好主意"然后直接用。不要"其实我已经想到了"——这句辩解让帮助变成了冒犯。

**展示测试意识。** 写完代码后，主动问一句"我可以用一个测试用例跑一下吗？"或者自己口述一个测试场景验证逻辑。这展示了你关注代码质量。

**不要过度解释。** 确定性的问题直接给出答案，不要绕圈子。越是确定的表达，越有专业感。

### Pairing 中的信号

- 不断问面试官"接下来我该做什么" → 没有主动性，依赖外界指令
- 面试官给了 bug 提示后直接找到 bug 的位置 → 好信号，展示 debug 能力
- 面试官给了 bug 提示后在同一位置反复犯错 → 红信号，说明没有从反馈中学习

## 五、自学者容易踩的坑

### 过度强调"自学"

在技术面中不要主动说"因为我是自学的所以不懂这个"——这既不能帮你加分，还会降低面试官对你的预期。

如果面试官问到你不会的东西，直接面对就好了。不需要用"自学的"来给自己找台阶——你过去几个月的经历，对方已经通过简历了解了。

### 过度防御 vs 过度顺从

这两个极端自学者都比较常见。

**过度防御：** 面试官给了一个 suggestion，你花一分钟解释"为什么我当时是这么做的"，潜台词是"你觉得有问题是你没理解我的场景"。

**过度顺从：** 面试官说一个方案，你立刻说"好的好的"，没有独立判断。

中间的姿态是：先接纳，再思考，然后回应。

"这个角度我确实没有考虑到。/ 嗯，你说得对，这样做确实更好。（先接纳）不过我在实际项目中遇到了 X 限制，所以当时选择了 Y 方案。你觉得这个场景下 Z 方案能解决吗？（再讨论）"

### Strong Hire vs Strong No 的分界线

面试官对候选人的心理评价大概经历三个阶段：

```
阶段 1：找理由接受（"这个候选人可以带"）
阶段 2：感觉到风险（"他这里好像有点问题"）
阶段 3：形成结论（"值得 / 不值得"）
```

软技能的优秀表现，会让你在阶段 1 停留更久。面试官会主动帮你找"这个人可以"的理由。反之，软技能的红灯信号会让你迅速滑向阶段 3。

## 六、软技能不是天生的，可以刻意练习

很多人觉得"沟通能力"是性格的一部分，改变不了。但技术面的沟通能力是**技能**，不是**天赋**。

刻意练习的方法：

**录自己能解决的问题。**

找一道你会的算法题，开录音讲一遍你的思路。听回放——你可能会发现你的表达远远没有你以为的那么清晰。

**模拟面试是最好的训练。**

找朋友或付费的模拟面试服务。技术面的软技能场景只有通过真实的面试氛围才能检验出来。自己在纸上写答案永远看不到"面试时的你"是什么样子的。

**面试复盘时重点记录软技能失误。**

每次面试后，除了记录"哪些技术题答错了"，还要记录"哪些问题我没追问"、"哪个 moment 我防御了"、"哪个地方我沟通不清晰"。

---

## 写在最后

技术面试到了某个水平之后，区分"通过"和"挂掉"的，往往不是技术深度差那一点，而是软技能的差距。

对于自学者，技术面的软技能是你的**隐藏杠杆**——技术深度可能需要时间积累，但沟通方式、问题拆解能力、面对反馈的态度，这些东西可以在短期内通过刻意练习大幅提升。

**抓住这个杠杆，你会发现面试通过率的提升远比想象中快。**
