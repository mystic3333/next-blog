---

对于自学转行的人，项目**就是你的工作经验**。

HR 不看你的 GitHub 星星数，面试官不会因为你的项目用了最新框架就给你加分。他们看项目的方式和你想象的完全不同。

这篇文章会从面试官的视角拆解：什么样的项目能加分、什么样的项目等于没做、以及怎么做项目才能让面试官觉得"这个人值得招"。

---

## 一、面试官怎么看你的项目

先做一个思想实验。你是面试官，现在有两个候选人：

**候选人 A**：做了一个"在线商城"，用了 React + Node.js + MongoDB。功能包括：商品展示、购物车、下单。代码 5000 行。

**候选人 B**：做了一个"团队日常开支记账工具"，用了 Next.js + TypeScript + PostgreSQL。功能包括：扫码录入账单、按成员分摊费用、月度统计图表。代码 3000 行。GitHub 上有 3 个真实用户的 issue 记录。

哪个项目更有说服力？

答案是 B。原因在于面试官看项目的四个维度：

### 1. 真实感

B 的项目解决的是一个真实问题——几个人一起吃饭谁付了多少钱怎么分摊。这个问题面试官自己也遇到过。而"在线商城"是培训机构的标准毕业项目，面试官已经看过几百个了。

**真实项目 = 有真实需求 + 有真实用户 + 有真实迭代。**

你不需要做一个改变世界的产品。只需要做一个你身边的人真的在用的东西。

### 2. 深度而不是广度

一个项目做了 10 个功能但每个都很浅，不如一个项目只做 3 个功能但每个都有技术深度。

什么是深度？

- 不是"实现了 CRUD"，而是"设计了一个缓存策略减少 60% 的数据库查询"
- 不是"用了 WebSocket"，而是"处理了断线重连和消息顺序保证"
- 不是"接入了支付"，而是"处理了支付回调的幂等性和对账"

深度 = 你在一个点上想得比别人多。

### 3. 可盘问度

面试官会在面试中深挖你的项目。他问的问题越深，说明对你的兴趣越大。

一个"深"的项目能经得起这样的追问：

- "为什么选这个数据库？为什么不选别的？"
- "这个方案有没有考虑过其他替代方案？"
- "如果用户量翻 10 倍，哪里会先崩？"

如果你的项目被问到第二层就答不上来，说明做的时候没想透。

### 4. 你的角色感

面试官想知道：这个项目里，你扮演了什么角色？

是"照着教程敲了一遍"？是"承担了某个模块"？还是"从 0 到 1 独立设计并实现"？

即使你是一个人做全栈，也要能讲清楚每个决策背后的思考过程。

## 二、哪些项目不要做

有些项目看起来高大上，但在面试中反而减分。直接避坑：

### ❌ CRUD 管理系统

"学生管理系统"、"图书管理系统"、"订单管理系统"——培训机构万年不变的作业项目。面试官看到标题就已经免疫了。

不是不能做 CRUD，而是所有项目的本质都是 CRUD。区别在于你的 CRUD 有没有附加价值。

### ❌ 纯 Demo 级项目

一个 Todo List、一个计数器、一个天气查询。这些只适合学习阶段，不适合放在简历上。

### ❌ 抄袭痕迹明显的项目

UI 跟某个知名产品长得一模一样，代码里有教程的注释没删干净，README 里还有"TODO: xxx"。这些都在告诉面试官：这不是你的项目。

### ❌ 太大而无法独立完成的项目

"做了一个电商平台"——面试官听到第一反应不是"好厉害"，而是"这肯定不是他一个人做的"。

宁可做一个完整的、小而精的项目，不要做一个半成品大项目。

## 三、什么样的项目值得做

### 类型 1：跨界项目（最推荐）

把你之前行业的经验和编程结合起来。这是你作为转行者的独特优势。

- 之前做销售 → 做一个客户管理 + 销售预测工具
- 之前做财务 → 做一个个人/小微企业记账 + 报表工具
- 之前做教育 → 做一个学习计划管理 + 进度追踪工具
- 之前做运营 → 做一个内容排期 + 数据看板

为什么这种项目最加分？因为面试官看到了一个能**用技术解决实际问题**的人，而不是一个"会写代码但在等其他人的需求"的人。

### 类型 2：开源贡献

给一个真实的开源项目提 PR，哪怕修复一个很小的 bug、改进一个文档。这说明你：
- 能读懂别人的代码
- 能遵循社区的协作规范
- 愿意为项目质量做贡献

### 类型 3：工具类项目

做一个解决特定痛点的工具。比如：
- 命令行工具：批量重命名文件、自动生成项目模板
- 浏览器插件：网页标注、快速搜索
- Slack/Discord Bot：自动提醒、信息聚合

工具类项目的好处是**边界清晰**，面试官很容易理解你在做什么，也容易验证它的价值。

## 四、怎么做项目才有深度

### 选一个有挑战的点

不要"均匀"地完成所有功能。在项目里选一个点深入钻研：

- 性能优化：数据量大的时候怎么保证流畅？
- 用户体验：Loading、空状态、错误处理、边界情况
- 安全性：XSS、CSRF、SQL 注入怎么防范？
- 可维护性：代码结构、组件设计、状态管理为什么这么选？

面试官想看到的是：**你会在一个点上比平均水平想得更深。**

### 写 README 和文档

很多自学者不做这一步，这是巨大的损失。一份好的 README 本身就是技术能力的展示。

它应该包括：
- 项目背景（解决什么问题，为什么做）
- 技术选型及理由（为什么选这个框架/数据库）
- 架构设计（目录结构、数据流、关键模块说明）
- 部署说明（CI/CD、Docker、环境变量说明）
- 待做清单（Next Steps）—— 说明你有产品思维

### 写测试

即使只是几个关键功能的单元测试，也说明你有质量意识。很多初级开发都不会写测试，你能写就是加分项。

---

## 五、面试中怎么讲项目

自我介绍环节讲项目 ≠ 把 README 背一遍。你在面试中讲项目的时间最多 3-5 分钟，所以要有结构。

### 推荐结构

```
背景（30 秒）：这个项目解决什么问题，为什么做
技术选型（30 秒）：用了什么技术栈，为什么选这些
我做了什么（2 分钟）：挑 1-2 个最有技术含量或最难的点展开讲
遇到的挑战（1 分钟）：讲一个真实的困难 + 你怎么解决的
学到了什么（30 秒）：这个项目让你在哪个方面有了明显的成长
```

### 准备好你的"单点叙事"

针对项目里的每个技术点，准备一个"TAG"结构：

- **T**ask：我遇到了什么问题/需求
- **A**ction：我做了什么方案调研和决策
- **G**ain：最后达成了什么效果（尽量有数据）

面试官问项目细节的时候，不要只回答问题本身，可以用 TAG 结构展开成一个完整的小故事。

---

## 六、项目迭代思维

一个好的项目应该被看作一个**持续迭代的产品**，而不是一次性的作业。

做完第一版之后，可以做这些事情让它变得更好：

**第一轮：功能完整**
- 完成核心业务逻辑
- 上线可访问（Vercel/Render/Railway 免费部署）

**第二轮：质量提升**
- 加单元测试和集成测试
- 优化首屏加载时间
- 处理错误边界和异常情况
- 补全 README 和文档

**第三轮：深度拓展**
- 加 CI/CD（GitHub Actions）
- 引入 Docker 部署
- 做一个小型性能测试并优化

每一轮迭代，都是你简历上多一行有说服力的内容。更重要的是，这个过程本身就在训练你**工程化的思维方式**——而这正是面试官最想看到的。

---

## 写在最后

项目是自学者简历上唯一能证明"我能写出可用的软件"的东西。不要把它当成作业来完成，把它当成你的作品来打磨。

一个精良的项目，足以让面试官忽略你没有科班背景和正式工作经验。

**记住：面试官不是在找一个"写过代码的人"，他在找一个"能解决问题的人"。你的项目就是证明你能解决问题最直接的方式。**
