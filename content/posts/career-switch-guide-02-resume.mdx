> 系列目录：本文是「自学转行面试指南」系列的第 2 篇。上一篇我们讲了转行的底层认知，这篇进入最实操的环节——写简历。

---

如果你在自学转行，你的简历大概长这样：

```
教育经历：XX 大学（非计算机专业）
工作经验：销售 / 运营 / 行政 / 其他非技术岗位
技能：JavaScript、React、Node.js（自学）
项目：xxx 管理系统、xxx 博客、xxx 电商 Demo
```

然后你投了 50 份简历，0 个面试邀请。

问题出在哪？**不是你没有能力，是你的简历没有呈现能力。**

---

## 一、简历的核心逻辑

先搞清楚一件事：HR 和面试官看简历的时间平均只有 **6-10 秒**。这 6 秒里，他只有一个问题：

> "这个候选人有没有可能胜任岗位？"

你的简历不需要证明"你很优秀"——它只需要让面试官觉得**值得给你一个面试机会**。

对于自学者来说，这意味着三件事：

1. **技能优先** — 不要藏着你自学的身份，也不要刻意强调它。把技能和经验放在前面
2. **用项目替代工作经历** — 没有工作经验不是缺陷，是你的简历结构需要调整
3. **故事线的一致性** — 你的简历要讲一个"为什么这个人从 A 转到了 B，而且他会在 B 领域做得很好"的故事

---

## 二、简历结构：自学者最有效的布局

不要用科班生的简历模板。你的优势区不同，布局也应该不同。

### 推荐结构（按顺序）

```
个人信息（姓名 / 联系方式 / GitHub / 博客 / LinkedIn）
↓
技术栈（核心技能清单）
↓
项目经历（2-3 个有深度的项目）
↓
工作经历（精简，一句话概括，突出可迁移能力）
↓
教育背景（一行带过）
```

### 为什么这么排？

科班生把教育背景放前面，因为那是他的优势。你把教育背景放前面，面试官第一眼就看到"非科班"，然后带着偏见往下看。

**控制面试官的阅读顺序，就是在控制他对你的第一印象。**

### 技术栈怎么写

不要写"了解"、"熟悉"、"精通"这种词——这几个词已经被用烂了，没有信息量。

推荐写法：

```
JavaScript（ES6+）| React（Hooks, Context, 自定义 Hook）| Next.js（App Router, SSR）
Node.js（Express, RESTful API 设计）| PostgreSQL（CRUD, 联合查询, 索引优化）
TypeScript（泛型, 类型推导）| Git（协作流程, 冲突解决）
```

关键点：
- **具体化** — 用括号说明你做到的深度，而不是泛泛说"熟悉"
- **跟 JD 对标** — 每投一家公司，根据对方的技术栈微调技能排序
- **诚实** — 面试官会深挖你写上去的每一项，不要写没做过的

## 三、项目经历：你的杀手锏

对于自学者，项目经历**就是你的工作经验**。它不只是展示你会写代码，更是展示你能独立交付完整功能。

### 一个项目应该怎么写

不要：

```
项目：个人博客
技术栈：React + Node.js + MongoDB
描述：一个可以发布文章的个人博客系统，支持 CRUD 操作。
```

这种写法在面试官眼里等于"我什么也没做"。因为你只写了"我做了什么功能"，没写"我怎么做的"、"解决了什么问题"。

要这样写：

```
项目：个人博客系统

技术栈：React + Next.js App Router + TypeScript + PostgreSQL + Supabase

关键实现：
- 基于 Next.js App Router 实现博客文章列表和详情页的 SSR 渲染，首屏加载时间降低至 1.2 秒
- 使用 Supabase 实现 GitHub OAuth 认证，支持用户登录后发表评论
- 设计并实现阅读量统计功能，基于 PostgreSQL 函数实现原子性的自增操作
- 集成 three.js 实现 3D 粒子背景，通过 dynamic import 将包体积减少 150KB
```

发现区别了吗？

- **有数据** — 1.2 秒、150KB 这些数字让面试官觉得你关注性能
- **有技术深度** — SSR、原子性操作、dynamic import 说明你不只是会用 API
- **有问题解决** — 每一行都能展开讲一个"我遇到了什么问题，怎么解决的"的故事

### 选什么样的项目

初级开发岗位，2-3 个有深度的项目足够了。选择标准：

1. **全栈项目** — 前后端都涉及，展示能独立交付完整功能
2. **有真实用户** — 哪怕只有你爸在用，也比没有强。真实用户意味着真实 bug 和真实需求
3. **解决真实问题** — 不要写"xxx 管理系统"这种烂大街的 Demo，写一个你生活工作中真的遇到的问题

### 项目从哪来

- 做一个你之前行业+编程的交叉项目（最推荐，展示你的跨领域优势）
- 给朋友/小公司免费做一个简单的官网或工具
- 参与开源项目，修一个 issue 就是一条经历

## 四、工作经历：把劣势变成优势

很多自学者会把工作经历写得很长，试图证明自己"不是全部从头开始"。

其实不需要。

你之前的工作经历在面试官眼里就两个作用：

1. **证明你是一个靠谱的成年人** — 有职场经验，知道怎么跟同事协作，不会干两个月就跑
2. **证明你有可迁移的能力** — 沟通能力、问题解决能力、自我驱动的能力

所以写法应该是：

```
XX 公司 | 销售经理 | 2019 - 2024
- 负责区域客户拓展，年度业绩达成率 120%，Top Sales
- 跨部门协调产品、运营团队推进项目落地
- 主导团队新人培训体系搭建，累计带教 10+ 新人
```

然后在面试的时候，把这些经历重新解释：

- "销售要理解客户需求 → 编程要理解产品需求，本质是一样的"
- "Top Sales 是因为我能快速学习产品知识 → 自学编程证明了同样的学习能力"
- "带教新人 → 我擅长把复杂的东西讲清楚，这种能力在写 code review 和技术文档时也一样"

**不要否认你的过去，要学会重新定义它。**

## 五、几个容易踩的坑

### ❌ 海投不改简历

每家公司都投同一份简历。这是效率最低的方式。花 15 分钟根据 JD 调整关键词排序和项目侧重，面试邀约率能翻倍。

### ❌ 技能造假

面试官会深挖简历上的每一项。写"熟悉 Docker"，结果 Dockerfile 都没写过，面试直接崩。宁可少写，别写假的。

### ❌ 简历太长

初级的岗位，一页纸足够了。多出来的内容不会加分，只会让人觉得你分不清重点。

### ❌ GitHub 没内容却放了链接

如果你 GitHub 上没有像样的项目，不如不放。空仓库比没有仓库更减分。

## 六、一份好的简历是什么样的

这是一份真实的、帮我认识的自学者拿到面试邀请的简历结构：

```
张三 | 前端开发（自学转行）
github.com/zhangsan | zhangsan.me

技术栈
React（Hooks、Context、自定义 Hook、性能优化）
TypeScript（泛型、类型工具）| Next.js | Tailwind CSS
Node.js + Express | PostgreSQL | Supabase

项目经历
在线协作白板工具 | React + Canvas + WebSocket
- 基于 Canvas API 实现自由绘图和图形编辑功能
- 使用 WebSocket 实现多人实时协作，延迟 <200ms
- 设计并实现 Undo/Redo 命令模式，支持 50 步操作回退

面试题库平台 | Next.js + TypeScript + Supabase
- 集成 AI 接口实现题目自动生成和答案评估
- 基于 next-auth 实现 Google OAuth 登录
- 使用 PostgreSQL 全文检索实现毫秒级题目搜索

工作经历
XX 教育公司 | 运营主管 | 2020 - 2024
- 负责用户增长策略，月活提升 40%
- 数据分析驱动决策，搭建用户分层运营体系

教育背景
XX 大学 | 市场营销 | 本科
```

这份简历透露出什么信息？

- 技术栈务实：React + TypeScript + Node.js，主流初级岗位的标配
- 项目有深度：一个项目聚焦一个核心难点（WebSocket / AI 集成 / Canvas）
- 项目有数据：延迟 <200ms、50 步、毫秒级 — 说明关注质量
- 工作经历简洁：几句话带过，但突出了可迁移能力（数据驱动、增长策略）

## 写在最后

写简历不是"如实记录经历"，是**用有限的空间，讲一个最有利的故事**。

你的故事是什么？一个目标明确、学习能力强、能独立交付完整功能的自学者。你的简历的每一部分都要服务于这个故事。

改完简历之后，还有一个关键步骤：**让别人帮你审简历**。不要只给自己看，发给同行、发到社群、发给你认识的任何做技术的人。别人的视角会告诉你"你写的东西我记得不够好"。
