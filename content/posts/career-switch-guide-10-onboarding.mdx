通过面试，拿到 Offer，这只是起点。

面试和真实工作之间有一条巨大的鸿沟：面试是**单点验证**，工作是**持续交付**。对于自学者来说，前 90 天尤其关键——你需要用这 90 天完成一件事：**建立信任**。

---

## 一、入职前两周 — 快速建立基础

### Day 1 要做的事

入职第一天，你不需要证明什么。

- 记下所有人的名字和职位。不要靠脑子记。
- 了解团队的技术栈和工作流——框架版本、代码规范、CI/CD 流程、Code Review 规范
- 按照文档搭开发环境。卡住了不是你的问题，是文档的问题。记录下卡住的地方，然后优化文档

### 第一周的目标

非常具体：

- 环境跑起来了
- 知道代码仓库的结构
- 能找到一个文件在哪里
- 跑通了一次完整开发流程（编码 → 提交 → 部署）

这个阶段不要试图理解所有代码。你只需要知道去哪里找什么。

### 第一周不要做的事

- 不要给代码提优化建议——你还不理解业务上下文
- 不要在下班后接着学——你会 burnout
- 不要沉默——遇到不懂的及时说，不要闷一天

## 二、如何提问

自学过程中你习惯了遇到问题先自己查。这是好事，但需要有边界。

### 15 分钟原则

**自己查的：** API 语法、框架配置项、开发工具用法、重现已知 Bug。

**问同事的：** 业务逻辑不理解、代码架构的设计意图、历史决策的原因、不确定团队规范。

### 正确的问法

错误的问法：「这个怎么弄啊?」

正确的问法：「我在做 XX 任务，遇到了 Y 问题。自己查了 A 和 B 两个方向，都走不通，分别出现 C 和 D 的错误。你能帮我看看方向对不对?」

前者把解决问题的责任甩给对方。后者展示了你尝试过，有具体上下文，对方只需要帮你确认方向。

别人给你的回答，记录下来。不要问第二次。

---

## 三、第一个项目 — 积累信任的关键

你的第一个任务是巩固信任印象的机会。

### 接任务的四步确认

不要只说「好的」。做四步确认：

1. **确认目标**：期望产出什么? 影响到什么用户?
2. **确认范围**：哪些这次做，哪些后续迭代?
3. **确认验收标准**：怎么算做完了? 有没有测试或 CR 标准?
4. **确认时间预期**：什么时候完成? 不确定的地方能不能提前对齐?

### 任务执行节奏

- 前 25% 时间：理解 + 设计。不急着写代码
- 中间 50% 时间：编码实现。先核心逻辑，再边界情况
- 最后 25% 时间：自测 + 文档 + 提 PR

提前跟 leader 对齐节奏。

### 主动汇报

每天下班前一条消息，三句话：

「今天完成了 X，明天做 Y，遇到了 Z 还在排查。」

一个主动同步进度的初级开发，比默不作声做完所有任务的人更可靠。

## 四、Code Review 生存法则

Code Review 是初级开发最容易受挫的地方。你写了一天代码，被人用 10 条评论打回来。这是正常的。

### 收到 Review 评论时

第一反应不要是「他是不是觉得我菜」。Review 是流程，不是人身攻击。

- 每条评论都回复。同意就说「改好了」，不同意就说理由
- 对于不理解的建议，直接问——「这个地方我没理解你的意思，能展开说一下吗?」
- 不要用「我觉得」「我个人认为」这种软化词——评论你的代码不是在质疑你这个人

### 自己的 PR 怎么写出少被 Review 的代码

- 提交前自己通读一遍 diff。自己能发现的低级问题，不要让别人来发现
- PR 不要太大。一个大 PR  reviewers 不想看，也看不细。拆成小的、独立的提交
- PR 描述写清楚：做了什么、为什么这么做、怎么测试的

## 五、自学者的独特优势

你来到这个岗位靠的不是科班基础，但你有一些科班生没有的优势：

**问题导向的思维方式。** 你学技术是因为要解决具体问题，而不是因为「这门课必修」。这意味着你更擅长把技术和业务需求连接起来。

**极强的信息筛选能力。** 自学过程中你必须分辨哪些资料值得看、哪些是浪费时间。这种能力在工作中同样可贵——面对陌生技术栈，你比科班生更快找到入口。

**对机会的珍惜。** 你走了更远的路来到这里，所以你比那些顺理成章进入这个行业的人更在意这份工作。这种在意会在日常中被同事感知到。

## 六、90 天的分阶段目标

### 第 1-30 天：融入期

- 能独立完成一个简单任务（修 Bug、做小功能）
- 知道了团队里谁是技术大牛、谁是业务专家、谁好说话
- 养成了主动同步进度的习惯

### 第 31-60 天：贡献期

- 开始承担中等复杂度的任务
- 有一次成功的跨团队协作经历
- 你的 PR 被 Review 的评论数量开始减少

### 第 61-90 天：独立期

- 能独立完成一个完整功能模块
- 开始参与架构设计讨论（至少能提出有质量的问题）
- 团队开始把任务直接分配给你，不需要 leader 转述

## 写在最后

面试系列到这里结束了。

10 篇文章，从认知到简历、从项目到投递、从面试到入职。这些方法论的背后只有一条主线：**用专业的态度弥补经验的不足。**

你不需要一开始就什么都会，但你需要知道怎么学、怎么问、怎么让别人信任你。

如果你是零基础自学转行，这条路确实不容易。但回头看看那些已经走通的人——他们跟你没有本质区别，只是比你早出发了一点。

**你已经在路上了。坚持走下去，终点就在那里。**
